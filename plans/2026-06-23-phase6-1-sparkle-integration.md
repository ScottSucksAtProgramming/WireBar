# Plan: Phase 6.1 — Sparkle Integration + Check-on-Launch

> Source PRD: PRD.md, user stories 61-62
> Source SPEC: SPEC.md (Updates: Sparkle framework, hosted via GitHub Releases)
> Design decisions: Q17 (Sparkle from v1), Q38 (updates are public, license gates features not updates), Q39 (.dmg distribution)

## Architectural decisions

Durable decisions that apply across all update-related phases:

- **Framework**: Sparkle 2.x via Swift Package Manager
- **Feed format**: Appcast XML hosted on GitHub Releases (public, no auth)
- **Signing**: EdDSA (Ed25519) signatures on updates — Sparkle 2's default. No DSA fallback.
- **Update policy**: Updates are public and ungated. The license gates features, not the update channel. All users (free and paid) receive updates.
- **Distribution**: `.dmg` files attached to GitHub Releases. `generate_appcast` builds the appcast XML from release assets.
- **Check behavior**: Automatic check on launch with Sparkle's standard update dialog. User can disable in Settings (Phase 6.2).
- **macOS compat**: Sparkle 2.x supports macOS 11+, well within WireBar's macOS 13+ target.

## What to build

A thin vertical slice that adds Sparkle as a dependency, configures it at app launch, and performs automatic update checks against a GitHub Releases-hosted appcast. The standard Sparkle update dialog handles download, verification, and installation — no custom UI needed for this phase.

### Steps

1. **Add Sparkle 2 as an SPM dependency** in `project.yml`. Pin to a stable 2.x release.
2. **Configure Info.plist** with Sparkle keys:
   - `SUFeedURL` — appcast URL (GitHub Releases raw URL, can be a placeholder until first release)
   - `SUPublicEDKey` — the EdDSA public key (generated by Scott, see instructions below)
   - `SUEnableAutomaticChecks` — `YES` (default on)
3. **Initialize `SPUStandardUpdaterController`** in `AppDelegate`. Sparkle 2 uses this as the main entry point. Create it early in `applicationDidFinishLaunching` so the automatic check-on-launch fires.
4. **Store the updater controller** as a property on `AppDelegate` so it stays alive and can be referenced by the Settings UI in Phase 6.2.
5. **Verify entitlements** — Sparkle 2 with hardened runtime needs `com.apple.security.network.client` (already present). No additional entitlements required since app sandbox is disabled.
6. **Build and launch** — confirm the update check fires (will get a "no updates available" or connection error to the placeholder URL, which is expected).

### Acceptance criteria

- [ ] Sparkle 2.x is added as an SPM dependency and builds successfully
- [ ] `SUFeedURL` and `SUPublicEDKey` are configured in Info.plist
- [ ] `SPUStandardUpdaterController` is initialized at app launch
- [ ] App launches without crashes and Sparkle's automatic update check fires
- [ ] No new warnings or errors in the build
- [ ] All existing tests (65+) continue to pass
- [ ] EdDSA public key is wired (placeholder acceptable until Scott generates the real key)

## Scott's blocking task: Generate EdDSA key pair

Before the real key can be wired, Scott needs to generate the Sparkle EdDSA key pair. Instructions:

1. **After Sparkle is added via SPM**, Xcode downloads the Sparkle package. Find the `generate_keys` tool:
   ```
   find ~/Library/Developer/Xcode/DerivedData -name generate_keys -type f 2>/dev/null
   ```

2. **Run it:**
   ```
   /path/to/generate_keys
   ```

3. It will output a **public key** (base64 string) and store the **private key** in your macOS Keychain automatically (under the name "Sparkle EdDSA Key").

4. **Copy the public key** and paste it into `Info.plist` as the value for `SUPublicEDKey`.

5. **Back up the private key**: Export it from Keychain Access (login keychain → search "Sparkle") to a secure offline location. If you lose this key, you cannot sign future updates that existing installs will trust.

The private key never goes in the repo — it stays in the Keychain on the build machine and in your offline backup.
